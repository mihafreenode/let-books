# Markdown у развоју производа

## Резиме

Markdown је практичан формат за чување намере производа, документације, водича и доказа у облику који се може прегледати, преносити, диффабле и лако повезати са токовима рада валидације.

## Зашто је Markdown битан

Markdown се често третира као погодност писања. У раду вођеном спецификацијама, то је корисније од тога.

Даје тимовима лак начин да задрже знање о производу близу спремишта без скривања унутар власничких алата или докумената које је тешко прегледати.

То је важно за:

- спецификације
- упутство за имплементацију
- документација о тренутном стању
- туторијали и вики странице
- изворне мапе и белешке о доказима

## У чему је Markdown добар

Markdown добро функционише када садржај треба да буде:

- читљиво у сировом облику
- контролисано верзијом
- лако прегледати у разликама
- може се повезати преко докумената
- преносив у различитим окружењима
- касније се може конвертовати у друге издавачке формате

Ово га чини посебно корисним за системе производа и документације којима је потребна следљивост.

## Оно што Markdown не ради сам по себи

Markdown је користан, али сам по себи не спроводи исправност.

Не може само по себи да гарантује:

- да је садржај актуелан
- да су потраживања добро добијена
- да имплементација одговара документацији
- да терминологија остаје доследна
- да су валидација или ЦИ правила заправо покренута

Зато Markdown треба да буде део ширег система, а не да се меша са целим системом.

## Markdown у току рада вођеном спецификацијама

У току рада вођеном спецификацијама, Markdown је највреднији када се налази унутар ланца ауторитета.

на пример:

1. спецификације дефинишу правила производа
2. упутство за имплементацију дефинише тактичка правила
3. Документи о тренутном стању објашњавају шта сада постоји
4. изворне мапе прате значајне јавне тврдње
5. валидација и ЦИ појачавају стабилна очекивања

Markdown је медиј који помаже да ти слојеви остану видљиви и прегледани.

## Зашто је ово важно у раду уз помоћ вештачке интелигенције

Системи вештачке интелигенције имају користи од јасног, приступачног контекста.

Markdown помаже јер је:

- лако прегледати
- лако цитирати
- лако се разликује
- лако држати у близини спремишта имплементације

То га чини јаким форматом за меморију производа.

Али АИ такође чини једно ограничење очигледнијим: ако је Markdown нејасан, застарео или контрадикторан, помоћник може да генерише брзо одступање од њега. Дакле, вредност не долази само из формата, већ и из одржавања садржаја.

## Markdown као упутство за агента

Неки Markdown документи могу изгледати као туторијали окренути људима, али такође функционишу као оперативне смернице за АИ агенте.

Ово је посебно корисно када документ чини више од објашњења теме и уместо тога обухвата:

- жељени редослед тока посла
- избор алата
- ограничења животне средине
- путеви ескалације
- анти-обрасци које треба избегавати

У тим случајевима, Markdown постаје више од објашњења. Постаје издржљив интерфејс између знања о спремишту и понашања агента.

## Let Books примери за вођење агента

### Ток посла за отклањање грешака за Андроид

`docs/android-debugging.md` је снажан пример.

Читљив је као људски водич, али такође даје агенту конкретна оперативна упутства као што су:

- дајте предност физичким Андроид уређајима у односу на емулацију радне површине за рад са камером и бар кодом
- дајте предност Playwright CDP прилогу над поновљеним покретањем претраживача
- разликовати убризгавање CDP у току извршавања од токова посла закрпе у стварном извору
- прво успоставите инфраструктуру за поновно учитавање када корисник затражи тестирање уживо путем телефона

Таква врста Markdown не описује само домен. Активно води ток посла и избор алата.

### Опрема и припрема околине

`docs/style-guide/publishing/ubuntu-tooling.md` је још један добар пример.

Чита се као документација за подешавање сарадника, али такође говори агенту који алати се очекују да постоје или би требало да буду преферирани за класе посла као што су:

- Playwright за верификацију корисничког интерфејса и снимке екрана
- Hunspell за проверу правописа
- ImageMagick и OCR алати за визуелне и лажне токове рада
- Алати у вези са ADB за отклањање грешака на Андроид-у

Ова врста Markdown помаже агенту да размишља о очекиваном ланцу алата уместо да нагађа на основу генеричких подразумеваних вредности.

## Практична импликација

Када пишете Markdown за спремиште које могу користити АИ агенти, помаже да се пише имајући на уму два читаоца:

- људски сарадник коме је потребно објашњење
- агент коме су потребна експлицитна ограничења тока посла и алата

Најбољи документи раде и једно и друго.

Они остају читљиви туторијали или водичи за људе, а истовремено су довољно специфични да усмеравају понашање агента на поновљиве начине.

## Let Books контекст

Репозиторијум Let Books већ користи Markdown за неколико слојева меморије производа, укључујући:

- `AGENTS.md`
- `AGENTS-Implementation.md`
- `README.md`
- РЕАДМЕ-ови и водичи у области докумената
- изворне мапе према `docs/sources/`

Ово је добар пример да се Markdown користи не само за прозу, већ и за координацију на нивоу спремишта, структуру јавног писања и праћење доказа.

## Практично правило

Корисно правило је једноставно:

Користите Markdown да бисте намјеру производа учинили видљивом, прегледном и повезивом. Немојте претпостављати да сам Markdown одржава ту намеру истинитом.

Истина и даље зависи од:

- јасан ауторитет
- активно одржавање
- дисциплина прегледа
- валидација тамо где је то практично
