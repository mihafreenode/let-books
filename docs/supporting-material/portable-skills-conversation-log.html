# From Prompts to Skills

Most teams start AI-assisted development with prompts.

That is natural. A prompt is the easiest unit of instruction: fix this bug, translate this file, review this code, explain this regex.

But prompts are fragile. They live in chat history, issue comments, pull request discussions, or somebody’s memory. They solve today’s task, but they do not automatically become tomorrow’s engineering practice.

The more durable unit is the skill.

A skill is not just a prompt. It is a reusable way of doing work. It contains guidance, examples, expected artifacts, validators, failure modes, and success criteria.

For example, a repository may already have a localization skill without calling it that. It may contain translation guidelines, terminology rules, parity checks, untranslated-text detectors, semantic review notes, and CI validation. Together, those pieces describe how localization should be done.

The same applies to code documentation. A repository may require comments around regexes, validators, heuristics, assumptions, and non-obvious intent. That is not merely “add comments.” It is a skill for preserving maintainer knowledge.

The important shift is this:

A prompt tells an AI what to do once.

A skill tells an AI how a team repeatedly performs a kind of work.

Guidelines are not passive notes. They are the human-readable part of a skill.

Tests are not just correctness checks. They are memory, written in executable form.

The goal is not to make AI autonomous. The goal is to make engineering practice reusable.

A mature AI-assisted repository should therefore not only contain source code. It should contain reusable skills: localization governance, content parity, validator design, documentation preservation, pull request knowledge artifacts, and release checks.

That is where AI becomes less like a clever assistant and more like a disciplined participant in an engineering system.

The AI becomes replaceable.

The skill remains.
