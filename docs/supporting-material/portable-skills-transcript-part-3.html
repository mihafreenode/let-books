## Objective

Determine which existing guidance should become part of portable skills and which guidance should remain repository-specific.

The long-term goal is for the repository to become a consumer of its own skills.

Repository documents should eventually contain primarily:

- repository adaptations
- repository-specific procedures
- references to skills

Portable guidance should have a single authoritative home inside the skills system.

## Audit Scope

Review all governance and guidance sources, including but not limited to:

- AGENTS.md
- AGENTS-Implementation.md
- governance documents
- contributor guidance
- localization guidance
- validation guidance
- documentation guidance
- workflow guidance
- review guidance
- architecture guidance
- onboarding guidance

Also identify any other documents that function as guidance, standards, policy, governance, or process documentation.

## Classification Model

For every identified rule, recommendation, requirement, standard, principle, workflow expectation, review expectation, or governance statement, assign one primary classification:

### Move To Skill

Portable guidance that should become part of a reusable skill.

### Keep As Repository Adaptation

Repository-specific guidance that depends on:

- repository structure
- implementation details
- file locations
- supported locales
- workflow names
- tooling choices
- deployment architecture

### Duplicate And Reference

Skill should become authoritative, but the repository should retain a localized adaptation or example.

### Remove As Redundant

Duplicated, obsolete, superseded, or unnecessary guidance.

## Rule-Level Analysis

Do not classify entire documents as a unit.

Analyze individual rules and guidance fragments.

A single document may contain rules belonging to multiple classifications.

## Gap Analysis

Identify:

- duplicated guidance
- conflicting guidance
- guidance without a clear owner
- guidance that appears in multiple places
- guidance that is implicitly assumed but not formally documented

Do not create new skills.

If recurring clusters appear that do not fit existing skills, record them as:

"Potential Future Skill Candidates"

without proposing implementation.

## Deliverables

Produce the following documents:

### migration-audit.md

For each document:

- rule
- classification
- proposed skill ownership
- rationale

### skill-mapping.md

For each existing skill:

- rules proposed for migration
- source locations
- duplication notes

### repository-adaptations.md

Repository-specific guidance that should remain outside skills.

Include rationale.

### migration-plan.md

Recommended migration sequence.

For each phase include:

- scope
- expected benefit
- dependencies
- migration risk
- validation approach

## Migration Strategy

Prefer:

1. Engineering Governance
2. Localization Governance
3. Documentation as Product
4. CI/CD Governance

unless audit findings justify a different order.

## Success Criteria

Success is not measured by how much content moves.

Success is measured by:

- single ownership of reusable guidance
- reduced duplication
- clearer separation between portable skills and repository adaptations
- a realistic migration roadmap

Planning and analysis only.

No repository modifications.
No file creation outside the requested audit deliverables.
No implementation work.
