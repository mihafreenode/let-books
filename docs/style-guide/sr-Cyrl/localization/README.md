# Локализација

## Сажетак

`docs/style-guide/localization/` дефинише како вишејезички садржај знања треба да буде преведен и повезан.

## Сврха

Одржавати преводе следљивим, правописно исправним, структурно усклађеним са канонским енглеским изворима и прегледљивим као дугорочни систем локализације, а не као једнократни преводилачки задатак.

## Структура

- правила за локализације
- правила за ID-јеве чланака
- правила за повезивање међу језицима
- правила за усклађивање терминологије
- правила за евиденцију AI прегледа
- benchmark примери за превођење помоћу LLM-а

## Основни принципи

- Енглески је канонски ауторски језик платформе знања осим ако документ изричито не наведе другачију политику.
- Преводи су испоруке прве класе, али наслеђују канонски ауторитет извора из енглеског оригинала.
- Локализација је шира од превођења. Укључује текст приступачности, дијаграме, снимке екрана, натписе, навигацију, метаподатке и управљање.
- Српски латиница и српски ћирилица су засебне локализације.
- Македонски, албански и словеначки морају да очувају правилну изворну ортографију и Unicode знакове.
- Превод који генерише AI дозвољен је само као фаза нацрта. Не замењује преглед.

## Захтеви за вишејезичне чланке

Сваки блог чланак објављен у овом репозиторијуму мора да прати ова правила:

### 1. Прво канонски енглески

Сви блог чланци морају прво да буду написани на енглеском. Преводи се изводе из канонске енглеске верзије. Енглески чланак је ауторитативни извор за референце на доказе и спецификације.

За блог frontmatter, `canonical_language` треба да остане `en` и у преведеној датотеци. Поље описује ауторитет извора, а не локализацију преведене датотеке.

### 2. Обавезни frontmatter

Сваки чланак (канонски и превод) мора у YAML frontmatter-у да укључи поље `language_variants` које наводи свих 12 језичких варијанти са релативним путањама:

```yaml
language_variants:
  - ../en/<article-id>.md
  - ../sl/<article-id>.md
  - ../hr/<article-id>.md
  - ../bs/<article-id>.md
  - ../sr-Latn/<article-id>.md
  - ../sr-Cyrl/<article-id>.md
  - ../mk/<article-id>.md
  - ../sq/<article-id>.md
  - ../de/<article-id>.md
  - ../it/<article-id>.md
  - ../fr/<article-id>.md
  - ../es/<article-id>.md
```

Подржане локализације: `en`, `sl`, `hr`, `bs`, `sr-Latn`, `sr-Cyrl`, `mk`, `sq`, `de`, `it`, `fr`, `es`.

### 3. Footer Other Languages

Сваки преведени чланак мора на крају да укључи footer одељак "Other Languages" који наводи све друге доступне преводе са изворним називом језика.

И канонски енглески чланци морају да укључе овај одељак ради паритета.

### 4. Локализовани source-map stub-ови

Сваки водећи енглески блог чланак мора да има одговарајући source map под `docs/sources/en/`. Свака неенглеска локализација мора да има кратки source-map stub под `docs/sources/<locale>/` који упућује назад на канонски енглески source map.

Source-map stub-ови нису пуни преводи. То су кратке датотеке са показивачем на енглески оригинал и језички прикладним контекстом.

### 5. LinkedIn деривативни пакети

Сваки водећи енглески блог чланак мора да има LinkedIn деривативни пакет под `docs/blog/en/linkedin/<article-id>.md` који садржи:

- објаву за ширу публику
- објаву за институције/библиотеке
- објаву за студенте/ученике
- оквир нити

То су само деривати за друштвене мреже. Не захтевају превођење.

### 6. HTML верзије чланака

Сваки блог чланак мора да има HTML верзију уз Markdown извор. HTML верзија прати исту структуру као постојећа HTML документациона страница (`docs/<locale>/index.html`) и повезана је из одговарајућег документационог индекса за тај језик.

HTML верзије се генеришу из Markdown извора стандардним конверзионим током рада (заснованим на pandoc-у) и морају да остану усклађене са садржајем у Markdown-у.

### 7. Захтеви за HTML објављивање

- `docs/<locale>/index.html` сваког језика мора да наводи блог чланке доступне на том језику са директним линковима ка HTML страницама чланака.
- HTML страница чланка користи исти образац заглавља, подножја, стилова и пребацивања језика као постојеће документационе странице.
- `<head>` мора да укључи исправне `hreflang` алтернативне линкове за свих 12 локализација.
- Атрибут `data-page-type` на `<body>` мора да буде постављен на `"article"`.

## Правила за source map

- Source map-ови живе у `docs/sources/<locale>/`.
- Енглески source map-ови су пуне мапе доказа које повезују тврдње са документацијом у репозиторијуму.
- Неенглески source-map stub-ови су кратке датотеке које референцирају енглески оригинал.
- Source map-ови се креирају само за водеће чланке, не за сваку мању објаву.

## Модел зрелости превода

Користите ове нивое када говорите о квалитету превода или га пратите.

- Level 0: source language only
- Level 1: machine translated draft
- Level 2: spellchecked
- Level 3: automated QA validated
- Level 4: community reviewed
- Level 5: native-speaker reviewed
- Level 6: professionally reviewed

Где је практично, пратите зрелост по локализацији и по категорији садржаја.

## Очекивања за локализацију у већем обиму

- Локализујте јавно усмерен садржај знања, не само UI стрингове.
- Локализујте метаподатке приступачности као што су alt текст, натписи, ARIA ознаке и језички метаподаци.
- Дајте предност дијаграмима генерисаним из структурираних извора како би преведене варијанте могле поново да се произведу.
- Документујте терминолошке одлуке, очекивања рецензената и путање за решавање спорова.
- Држите валидацију и CI близу токова објављивања како би регресије локализације биле третиране као питања квалитета производа.
- Сачувајте репрезентативне AI примере прегледа како би суптилне исправке изворних говорника могле да информишу обуку, управљање и евалуацију.
